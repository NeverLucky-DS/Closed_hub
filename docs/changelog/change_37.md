# change_37

## Что сделано

- После вычисления **SHA-256** содержимого документа в `handle_document` вызывается **`find_active_file_by_sha256`**: если есть запись со статусом не `deleted` / не `cancelled`, загрузка **прерывается** — файл на диск не пишется, строка в БД не создаётся, пользователю в Telegram приходит пояснение с id записи и статусом.
- Индекс **`idx_files_sha256_active`** (partial по `sha256` для «живых» статусов) — в `schema.sql`, миграция **12** в `schema_patch.py`.

## Почему SHA-256, а не «похожесть»

- Детерминированное совпадение **байт в байт**; без ложных срабатываний и без тяжёлого сравнения контента. Похожесть текстов/PDF — отдельная задача (симхэш, fuzzy).

## Что можно улучшить

- При гонке двух параллельных загрузок одного файла теоретически возможны две строки до появия уникального ограничения; при необходимости — partial **UNIQUE** на `sha256` после очистки дублей в БД.
