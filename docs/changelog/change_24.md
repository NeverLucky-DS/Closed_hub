# change_24

## Почему «в ТГ есть, а в базе / на сайте странно»

1. **Запись в `events` только из лички с ботом**, когда маршрутизация даёт intent `event` (текст/пересылка обрабатывается как анонс). Сообщения **только в супергруппе** (без сценария через бота) **в таблицу не попадают** — пересылка в ленту через `ml_forward` шлёт пост в тему, но **не создаёт строку в `events`**.
2. **Раньше** `INSERT` выполнялся **после** синхронного `summarize_event` (Mistral). Пока нейросеть отвечала, строки в БД не было → на сайте пусто, хотя пост в группу уже мог уйти.
3. Заголовок «[пересланное сообщение]» шёл из префикса в тексте, если не успевал дедуп/саммари — на сайте добавлен разбор префикса в `_event_header` и fallback-заголовок при вставке.

## Что сделано

- **`MISTRAL_API_KEY_FOR_SITE`** (`mistral_api_key_for_site` в `config.py`): для **`dedup_event`** и **`summarize_event`** используется этот ключ, если задан; иначе — основной `mistral_api_key`.
- **Очередь саммари** (`services/event_summary_worker.py`): один воркер на процесс бота, `asyncio.Queue`. После успешного дедупа и **`insert_event`** задача **`enqueue_event_summary(application, event_id, raw_text)`** — Mistral в фоне обновляет `ai_summary` и при необходимости **`normalized_title`**.
- **Ответ бота** пользователю: явно, что запись в базу сразу, саммари «догонит».
- Воркер стартует в **`post_init`**, в **`post_shutdown`** отменяется до закрытия пула.

## Веб и «API новостей»

Связанные с объявлениями маршруты:

| Маршрут | Назначение |
|--------|------------|
| `GET /feed`, `GET /today` | HTML, чтение `events` из PostgreSQL |
| `GET /api/events/{id}/cover` | Файл обложки из storage |
| `PATCH /api/events/{id}/ends_at` | Дата окончания актуальности |

Создания новости через веб-API **нет** — источник правды для новых строк **`insert_event` в боте**.

- Персистентная очередь (Redis / таблица `jobs`), если воркер перезапускается часто и теряются задачи в памяти.
